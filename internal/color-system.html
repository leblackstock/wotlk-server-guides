Rules at a glance

1

Spec leads

The active specialization colors navigation, selected states, major dividers, and page-level emphasis.

2

Class remains visible

Paladin pink stays on class badges and small class-level markers instead of dominating every Paladin page.

3

Status is fixed

Success, warning, danger, and information never change when the class or specialization changes.

4

Mechanics stay local

Rotation groups, pull types, and market categories receive scoped tokens instead of borrowing status names.

5

Color supports labels

Text, icons, or component shape must carry the meaning. Color is never the only instruction.

Dark-mode foundations

Surface tokens are named by purpose rather than brightness. Card surfaces remain darker than their containing section, preserving the compact layered appearance already used by the guides.

Semantic status colors

These meanings are global. A Protection information note stays cyan; a Holy warning stays amber; a Retribution danger state stays danger red.

Success / safe

Confirmed requirements, completed checks, and safe states.

Warning

Caution, a meaningful tradeoff, or something that needs verification.

Danger

Failure risk, emergency response, or a harmful correction.

Information

Neutral server notes, supporting explanation, and reference information.

Implementation reference

Activate a guide theme

Load the token sheet before component styles and identify the guide on its body element.

<link rel="stylesheet" href="../assets/guide-color-system.css">
<link rel="stylesheet" href="../assets/style.css">
<body data-guide-class="paladin" data-guide-spec="protection">

Consume aliases, not raw spec colors

Component CSS should respond to the active theme without knowing which specialization selected it.

.guide-link {
  color: var(--theme-accent-soft);
  border-color: var(--theme-accent);
  background: rgba(var(--theme-accent-rgb), .08);
}
